# Production challenge questions and answers

These are defense answers, not claims that every production feature is already
implemented. A strong answer has four parts: business invariant, current design,
evidence, and honest next production step.

## Architecture and workflow

### Why is this not just a chatbot?

The LLM is one bounded component inside a stateful enterprise workflow. Sensor
ingestion, detection, classification, priority, authorization, persistence,
CMMS translation, idempotency, audit, cost enforcement, and evaluation are
ordinary code. The LLM has only three read-only tools and produces a draft that
cannot leave `pending_review` without a named human.

### Why separate detection, classification, and explanation?

They have different failure and latency requirements. Detection runs every few
seconds and must be cheap/deterministic. Classification uses numeric window
features and rejects unsupported inputs. Explanation requires language and
retrieval, where an LLM is useful. Combining all three in one prompt would make
continuous monitoring expensive, less testable, and harder to govern.

### Why does the classifier override a conflicting LLM?

The narrow classifier owns only a supported numeric decision and has explicit
schema/OOD gates. Retrieval can surface a superficially similar work order and
mislead the LLM. The classification guard retains a concrete in-distribution
verdict, removes inconsistent citations, records the rejected draft, and still
requires human verification. The LLM remains responsible for supported prose,
not the numeric class.

### Why retain both case trace and audit trail?

They answer different questions. Trace answers “how did this agent run reach
its draft?” Audit answers “who changed business state, when, and what downstream
effect occurred?” A tool trace is not a legally or operationally sufficient
business audit; an audit event alone does not explain model reasoning.

### Why one case per machine event rather than one per breached signal?

A physical bearing fault changes several correlated signals. Separate cases
would multiply planner work and paid calls while fragmenting evidence. The
detector emits one anomaly and stores all signals in context. Machine-level
cooldown/open-case dedup suppresses the rest unless a human deliberately cues a
fresh scenario.

### Why is priority not generated by the model?

Priority affects operational sequencing and safety. A transparent formula
using criticality, severity, recurrence, and safety can be reviewed and changed
as policy. The model may propose one clamped notch with justification; P1 cannot
be downgraded. A human owns the final decision.

### Why is every case reviewed if the confidence is high?

Confidence is an uncertainty signal, not authorization. The challenge forbids
autonomous machine action, and maintenance work can affect safety and
production availability. Abstention changes how uncertainty is presented; it
does not remove mandatory review from accepted cases.

## Telemetry and data engineering

### How does data enter today?

The deployed demonstration has two in-process sources. The simulator writes
Gaussian healthy readings plus controlled fault ramps. The replayer writes one
curated SKAB/CWRU row per tick. Both persist `TelemetryReading` and call the same
detector. A plant deployment would replace the source with a historian/broker
adapter while preserving the reading/catalog boundary.

### Is this really real-time?

It is a near-real-time demonstration: three-second backend ticks and browser
polling every few seconds. It is not a high-frequency streaming architecture.
Production would ingest through MQTT/Kafka/IoT Hub, partition by asset/tag,
process events idempotently, and push case updates rather than polling.

### Why store signal values as JSON?

It supports different sensor rosters in a one-week slice and keeps the catalog
as the schema authority. The trade-off is weaker indexing/compression and more
application parsing. At scale I would use a time-series/historian integration
or tag/value/partitioned model with schema registry and retain the current
canonical feature contract.

### Why use median/MAD instead of mean/standard deviation?

An excursion can drag its own mean and standard deviation, weakening detection.
Median/MAD are robust to outliers. The detector excludes the freshest excursion
points from its baseline and requires three sustained breaches. A flat-MAD
series falls back to standard deviation; no spread yields zero robust-z.

### How do you prevent label leakage?

Replay labels are removed from signal dictionaries and passed only to anomaly
storage/evaluation fields. Agent tools return machine catalog, telemetry values,
and maintenance records—never `ground_truth_fault`. Training and validation
hold out whole physical experiments, not random windows from the same run.

### What happens when a replay jumps between operating regimes?

The cursor retains 45 healthy rows before the label and suppresses detection
for the first 30 rows to refill the rolling baseline. Without warmup, the
artificial file jump would look like a machine anomaly. This protects evaluation
from a discontinuity introduced by the demo harness.

### Why retain telemetry for only 24 hours?

Continuous readings dominate storage and current detection needs minutes of
history. Cases snapshot their evidence, while business records/audit are not
pruned. Twenty-four hours is a free-tier demo setting, not a universal policy.
Production retention should follow customer regulation, historian ownership,
partitioning, archival, and investigation requirements.

## ML, OOD, confidence, and evaluation

### Is the trained classifier genuinely finished?

It is finished for a narrow supported decision: SKAB suction versus discharge
restriction. It uses 510 windows from 17 training experiments, grouped
out-of-fold calibration, a frozen three-experiment restriction test, and
novelty/schema gates. It is not a general plant fault model.

### Why Extra Trees?

The dataset is small, tabular, nonlinear, and made of engineered window
statistics. Extra Trees is fast, works well on nonlinear feature interactions,
does not require sequence-scale training infrastructure, and is inspectable
enough for a narrow layer. The selection was based on grouped validation, not
model fashion.

### What makes OOD detection genuine?

It is not only “confidence below a threshold.” The artifact contains an
IsolationForest with a threshold derived from leave-one-experiment-out
in-distribution novelty scores. Inference also requires the exact trained sensor
roster and minimum calibrated class confidence. Same-roster novel faults and
unsupported CWRU schema are rejected through different observable gates.

### Does AUROC 1.0 prove OOD is solved?

No. It is an internal result on 14 same-roster non-restriction SKAB contexts.
The sample is small and from one testbed. It proves that this implementation can
learn and apply a novelty boundary on the development corpus; it does not prove
cross-site OOD performance.

### Why calibrate LLM confidence deterministically?

Self-reported confidence tracks fluency and did not transfer reliably to real
data. The calibrator makes evidence dependencies explicit: precedent strength,
specific versus non-diagnostic language, and classifier agreement/abstention.
It is a pragmatic evidence gate, not a statistically final customer calibrator.
ECE measures the resulting mismatch on the held-out suite.

### What is the difference between overall accuracy, coverage, and selective accuracy?

Overall accuracy counts every trial, including abstentions as not correct.
Coverage is the fraction that names a class. Selective accuracy is correctness
only among those accepted answers. On real data the hybrid classifier is 7/8
overall, covers 7/8, and is 7/7 on accepted cases. Saying only “100%” would hide
the abstention and tiny denominator.

### What does ECE 0.148 mean?

Across confidence buckets, the weighted average absolute difference between
stated confidence and observed accuracy was about 14.8 percentage points. It is
not accuracy and is based on eight cases, so it has high sampling uncertainty.

### Why two scorers?

The free-text scorer maps root-cause language; the independent citation scorer
maps cited work-order ids. Agreement checks the measuring instrument. Joint
coverage is reported so 100% agreement cannot hide that both scorers only
graded a few rows. Low synthetic live citation coverage is a measurement and
citation-quality warning.

### Why only eight real episodes?

The assessment was a one-week executable slice. Eight episodes across two
testbeds are more credible than synthetic-only evidence but insufficient for
production claims. The next evidence step is natural customer fault-onset data,
site-level external holdout, confirmed data rights, and prospective human
outcomes—not repeating the same eight recordings.

### Why include CWRU if it maps three fault subtypes to bearing wear?

It tests a genuinely different testbed and sensor schema, exercises ingestion,
relative detection, coarse taxonomy mapping, and schema-OOD behavior. It does
not claim subtype diagnosis; inner-race, ball, and outer-race are intentionally
mapped to the product's coarse actionable bearing family.

### Does synthetic performance matter?

It validates repeatability, detector coverage, complete workflow plumbing, and
known signature behavior across controlled classes. It is not evidence of field
generalization. Real replay and honest limitations are reported separately.

## LLM behavior and safety

### What exactly can the LLM do?

It can request machine information, recent telemetry, and maintenance-history
search, then draft root cause, confidence, explanation, actions, citations, and
a bounded priority adjustment. It cannot control machinery, approve a case,
switch its own paid mode, write the CMMS, or access evaluation labels.

### Why maintain a deterministic mock?

It makes the complete workflow demonstrable without provider cost/network
dependency, provides a stable baseline, and is the degraded mode on provider or
budget failure. It is not a claim that scripted retrieval is an LLM.

### What if OpenRouter times out or rejects the key?

The reserved ledger row is marked failed; trace records the provider exception;
the agent restarts from the original system/user messages using mock and still
creates a case. Anomaly evidence is not discarded.

### What if the model sends malformed tool arguments?

The orchestrator never guesses or repairs intent. It records
`invalid_tool_arguments`, sends a structured error to the model, and allows a
retry within the eight-turn bound. If no valid final answer arrives, it creates
a low-confidence human-escalation case.

### What if the model never converges?

After eight turns, the case states that the agent did not converge, confidence
is low, and manual triage is recommended. Bounded loops control latency/cost.

### Is maintenance history a prompt-injection risk?

Yes. Tool data is untrusted external content. Current tools are read-only and a
human remains mandatory, which limits consequences, but production should
structure/delimit retrieved fields, apply content policies, prevent tool-output
instructions from changing authority, validate final actions/citations, log
provenance, and test adversarial work-order text. The current artifact does not
claim full prompt-injection hardening.

### Could the LLM fabricate a work-order citation?

It can emit an id, but evidence retains retrieved matches and the classification
guard can clear conflicting citations. Production should additionally enforce
that every final cited id appears in tool results before persisting it. Current
scorer coverage makes citation quality visible; strict citation allowlisting is
a reasonable next hardening step.

### Why DeepSeek rather than GPT-4o?

DeepSeek V4 Flash passed the deliberate tool-use evaluation at low exact cost.
The architecture keeps model selection configurable and compares behavior
through evaluation rather than brand preference. GPT-4o mini is a candidate
fallback; full GPT-4o is not the cost-first choice.

## Integration and reliability

### Is the CMMS actually separate?

It is a separate FastAPI application and HTTP/domain contract, but it is mounted
in the same Render process and shares the database for the demo. The schema,
validation, headers, status handling, retries, and idempotency are genuine. A
real SAP/Maximo integration needs independent deployment/networking,
authentication, customer mappings, and contract tests.

### Why call the adapter an anti-corruption layer?

Internal code speaks machine id, P1–P4, root-cause text, and case evidence. The
CMMS speaks equipment/functional location, priority codes 1–4, M1, damage codes,
short/long text, and system status. Only the adapter knows both vocabularies, so
foreign schema does not leak into core models.

### How do retries avoid duplicate work orders?

Every approved case uses `Idempotency-Key: triage-case-{id}`. The CMMS table has
a unique key and returns the original work order on repeats. Transport/5xx
retries are therefore safe. A 4xx is a deterministic translation error and is
not retried.

### What if CMMS is down during approval?

The human decision is committed and audited before sync. Exhausted transient
retries mark `cmms_sync_status=failed`, and an authenticated endpoint retries
later with the same idempotency key. Rejection creates no work order.

### What if the process crashes after approval but before sync status is written?

The approved decision remains durable, but the current demo has no transactional
outbox/reconciler and the UI hides a blank sync status. Production should write
an outbox row in the same decision transaction, have a durable worker deliver
it, and reconcile all approved unsynced cases. This is a known gap, not a lost
decision.

### What if the process crashes after anomaly creation but before case creation?

The anomaly row is durable, but an id waiting only in the process queue is not.
The worker catches ordinary exceptions, yet a process death needs a recovery
scanner or durable broker that claims `status=new` anomalies idempotently. That
is the correct production extension.

### Is the current triage queue durable?

No. It is an asyncio in-process queue created to decouple telemetry timing from
30–60 second LLM calls in the single-worker demo. Production should use SQS,
Kafka, Celery/RQ, or a Postgres-backed job/outbox with retries, visibility,
dead-lettering, idempotent case creation, and lag metrics.

### Why one serial triage worker?

It preserves order, prevents provider bursts and budget races, and is sufficient
for intentional demonstration faults. It is not high-throughput. Production
would use bounded concurrency partitioned by machine/site, persistent request
reservation with row locks, provider rate limiting, and backpressure.

### What happens if the database is unavailable?

Request/feed operations fail; the simulator loop logs and continues trying on
future ticks. There is no local durable buffer, circuit breaker, or alerting.
Production ingestion would acknowledge only after durable broker/database
write, retry with bounded backoff, surface health degradation, and page on lag
or write failure.

### Is GitHub keep-warm monitoring?

No. It is a best-effort health request every ten minutes to reduce free-tier
cold starts. It does not inspect business SLOs, queue lag, detector silence,
case latency, provider failures, database health, or CMMS backlog. Production
needs real metrics, logs, traces, alerts, and synthetic workflow checks.

## Scalability

### What breaks first at 10,000 machines?

The in-process simulator/replay state, synchronous per-reading commits,
single-worker triage, JSON telemetry storage, polling UI, substring history
search, and Python-side audit filtering. The domain boundaries remain useful,
but the infrastructure must become event-driven and horizontally partitioned.

### How would you scale ingestion?

Use customer historian/OPC-UA/MQTT connectors -> durable broker -> schema
validation/normalization -> partitioned detector workers keyed by machine ->
time-series storage. Assign event ids and idempotent writes, preserve event time,
handle late/out-of-order data, track watermark/lag, and separate raw retention
from case evidence.

### How would you scale triage?

Publish anomaly ids to a durable queue. Use bounded stateless workers, one active
job per anomaly, unique constraint on case/anomaly, provider concurrency limits,
database-atomic budget reservation, retry/dead-letter policies, and queue-lag
SLOs. Cache/reference machine/history data safely and batch retrieval where
possible.

### Can you run multiple backend replicas today?

Not safely for feeds. Every replica would have its own simulator/replay cursor,
force set, LLM override, queue, and secondary budget counter, potentially
duplicating telemetry and disagreeing on mode. Production needs one elected feed
owner or external ingestion, shared configuration/state, durable queues, and
database concurrency controls before horizontal replicas.

### Would the paid-call cap hold under concurrency?

The DB ledger is persistent, but check-then-insert is not locked across multiple
processes. The code itself notes that multi-worker deployment requires a row or
advisory lock. Production should reserve against a single daily-budget row in a
serializable/locked transaction or use an atomic metering service.

### How would the frontend scale?

Stop polling complete fleet/case lists. Use paginated/filterable server queries,
server push/subscriptions for case state, virtualized fleet views, cached static
catalog data, and tenant/site authorization. The current three-query fleet
snapshot already avoids an N+1 pattern but is not a 10,000-card UI.

### How would maintenance retrieval scale?

Current retrieval loads same-type rows and performs Python substring scoring.
Production should query the real CMMS/search index using structured filters,
full-text/embedding hybrid retrieval, access controls, recency/equipment boosts,
record-type filters, citation allowlisting, and retrieval-quality evaluation.

### How would the database schema evolve?

Replace `create_all` with versioned Alembic migrations, backward-compatible
expand/migrate/contract releases, migration tests, backups/PITR, indexed foreign
keys/status/time columns, partitions for telemetry/audit, and environment change
control. `create_all` is acceptable for a demonstration, not governed schema
evolution.

## Security and governance

### Is the current authentication production-grade?

No. It provides a signed reviewer identity and protects state-changing routes
with a crew password, which is enough to demonstrate accountability. Production
needs SSO/OIDC, RBAC by site/role/action, short-lived/revocable sessions,
central audit, secrets rotation, MFA policy, and separation of operator/admin
permissions.

### What are the current browser-auth risks?

The token is stored in localStorage, so XSS could steal it. The demo has no
third-party script surface, but production should use secure HttpOnly SameSite
cookies or managed identity patterns, CSP, CSRF controls where applicable,
dependency scanning, and output encoding/security tests.

### Why is `ALLOWED_ORIGINS=*` a limitation?

It lets any origin call public endpoints and attempt authenticated requests.
Bearer-token semantics limit some CSRF risk, but production should allow only
approved frontend origins, define methods/headers narrowly, and combine this
with identity and network controls.

### Where are secrets stored?

Database password, crew password, and OpenRouter key are Render environment
secrets and are not committed or sent to the browser. Vercel receives only the
public API base URL. Production should use a secret manager, rotation, scoped
service identities, access audit, and no shared crew secret.

### Is there tenant isolation?

No. The application models one demonstration fleet. A multi-customer product
needs tenant/site ids on every row, row-level and application authorization,
tenant-scoped encryption/keys where required, isolated retrieval, data residency
policy, per-tenant budgets, and tests proving no cross-tenant access.

### Can a user modify or delete audit events?

There is no application endpoint to mutate audit rows, but the database table
is not an external immutable/WORM log. Production should ship signed/append-only
events to a restricted audit store/SIEM with retention and tamper controls.

### What about data privacy and regulation?

The current data is machine telemetry and demo reviewer names, not personal
customer data. Production still needs data classification, minimization,
retention, residency, vendor processing agreements, encryption, access review,
incident response, and dataset/license clearance. Maintenance notes can contain
personal or sensitive operational information and must be governed.

## Business and customer delivery

### How does this reduce unplanned downtime?

It reduces time from anomaly to an evidence-backed, prioritized, system-of-
record-ready ticket; it does not prove avoided downtime yet. The business pilot
should measure detection-to-review time, planner acceptance/edit rate, false
alerts, time-to-work-order, mean time to repair, repeat failure, and confirmed
downtime avoided against a baseline.

### Are the dollar exposure values real?

They are transparent illustrative asset-catalog inputs multiplied by downtime
from cited precedents or a four-hour fallback. They make business impact visible
but do not enter the priority formula. A customer rollout replaces them with
finance/operations-approved line rates and validates attribution.

### How would you onboard a customer plant?

Start with workflow discovery: who owns alarms, triage, approvals, and CMMS.
Inventory tags/units/sample rates and map them to canonical roles. Connect a
historical read-only source, profile missingness/regimes, agree engineering
limits/taxonomy/priority policy, map CMMS contracts, run shadow mode, collect
confirmed outcomes, calibrate by site, complete safety/security review, then
roll out by asset class with acceptance/SLO gates.

### What would you ask the maintenance planner?

- Which alerts currently create work and which are ignored?
- What evidence is required before dispatch?
- Which faults are separable from existing tags?
- What is the cost/safety consequence of false positive versus false negative?
- How are priority and escalation actually decided?
- Which CMMS fields/codes/approvals are mandatory?
- How is the true failure confirmed after inspection?
- What latency and uptime matter by asset criticality?
- Who can approve, edit, reject, and retry?
- What data may leave the plant or reach a model provider?

### What is the single best product extension?

A closed-loop outcome and shadow-evaluation system: on work-order completion,
record confirmed fault/action/downtime/usefulness; join it to detector,
classifier, LLM and human decisions; measure overrides, selective accuracy,
time/cost impact, and drift; run challengers silently; promote only after enough
external outcomes and no safety regression. It creates a learning system
without uncontrolled self-modification.

## Incident and debugging defense

### Explain the 2026-07-19 delayed/duplicate-case incident.

A CWRU cue at 21:25:38 IST was not detected until 21:32:18 because the telemetry
loop synchronously awaited slow LLM triage. The multi-signal bearing episode
then created four anomalies/cases—one each for RMS, kurtosis, RPM/crest changes—
and several paid tool turns reached the 12-request guard. Audit timestamps were
correct; processing was late.

The fix separated telemetry ticks from a serial triage queue and changed dedup
from metric to machine-event level, breaking after the first anomaly while
retaining every signal as context. Tests now assert that slow triage cannot
pause ticks and a multi-signal event creates one anomaly. Values and audit
timezone/progress messages were also clarified.

### Why did 12/12 not mean $0.25 was spent?

The primary ceiling counts provider requests, and each tool-calling case uses
several completion turns. The incident ledger recorded $0.003425; the request
cap, not the dollar cap, stopped further live calls. This is why the UI shows
both calls and exact returned cost.

### What did this incident demonstrate about your engineering process?

The audit trail enabled exact cue/anomaly/case timing reconstruction. Code
inspection identified synchronous orchestration and per-metric dedup as two
separate causes. Regression tests captured both invariants, deployment exposed
a release id, and UI/docs were updated so operators could distinguish waiting,
cost, and timezone behavior. The honest remaining gap is queue durability.

## Fast challenge answers

| Challenge | Short answer |
|---|---|
| “Why not an LSTM autoencoder?” | Possible detector candidate, but current deterministic rules are explainable and adequate for the slice; compare models on external event-level labels before replacing them |
| “Why no vector database?” | Seeded history is small and keyword ranking is auditable; real-scale hybrid retrieval is an adapter evolution, not a reason to add infrastructure theater |
| “Why not let high confidence auto-approve?” | Confidence is not authority; assessment and maintenance risk require a named human |
| “Is isolation forest calibrated probability?” | No; it is a novelty score with a held-out-derived threshold, separate from calibrated class probability |
| “Is 7/7 statistically strong?” | No; always pair it with 7/8 overall and n=8 |
| “Does the agent see the label?” | No; label is evaluation-only and absent from all tools/prompts |
| “Does keep-warm cost LLM money?” | No; it only GETs health |
| “Can approval create two orders?” | Same case idempotency key returns the existing order |
| “Does rejection call CMMS?” | No; it sets not-applicable and stops |
| “Can the LLM downgrade safety P1?” | No; deterministic adjustment refuses a P1 downgrade |
| “Can this control a machine?” | No tool or endpoint performs machine control |
| “Is this SAP-ready?” | It demonstrates the anti-corruption contract; production SAP auth/API/mapping/contract testing remain |
| “Is Render production infrastructure?” | It is deployment proof for the assessment, not the target plant SLA architecture |

## Final defense principle

Do not answer every question by listing technology. State the operational
invariant first:

> I need to preserve the anomaly, make uncertainty visible, keep authorization
> human, prevent duplicate downstream work, and leave enough evidence to explain
> every state change.

Then show which component enforces it, what is tested, and what changes at
customer scale.
